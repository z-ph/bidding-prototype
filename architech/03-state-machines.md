# 03 - 状态机设计

> 每个核心业务对象必须有状态机。状态决定：可见页面、可执行操作、操作人、下一状态。
> 这是「只走主路径」问题的直接对策——补全生命周期分支。

---

## 1. Package（标段）状态机 - 主流程

标段是流程主体，状态机覆盖全生命周期。

```
                        ┌──────────┐
                        │  draft   │ 草稿（编辑招标文件）
                        └────┬─────┘
                             │ 发标（招标人/代理）
                             ▼
                        ┌──────────┐
                ┌───────│ bidding  │ 招标中（公告已发布）
                │       └────┬─────┘
                │            │ 报名截止
                │            ▼
                │       ┌──────────┐
                │       │registering│ 报名审核中（若需审核）
                │       └────┬─────┘
                │            │ 审核完成 / 无需审核
                │            ▼
                │       ┌──────────┐
                │  ┌────│ bid_open │ 投标中（可缴费/上传/报价）
                │  │    └────┬─────┘
                │  │         │ 投标截止
                │  │         ▼
                │  │    ┌──────────┐         询比价路径
                │  │    │pre_open  │────────▶│ compare  │ 比价大厅（可多轮报价）
                │  │    │ 待开标   │         │ 比价中   │
                │  │    └────┬─────┘         └────┬─────┘
                │  │         │ 开标时间到          │ 确定供应商 → pre_eval（或直接 awarded）
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │ opening  │ 开标中        │
                │  │    │ 解密/唱标 │               │
                │  │    └────┬─────┘               │
                │  │         │ 结束开标             │
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │pre_eval  │ 评标准备      │
                │  │    │ 抽取专家  │               │
                │  │    └────┬─────┘               │
                │  │         │ 启动评标             │
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │evaluating│ 评标中        │
                │  │    └────┬─────┘               │
                │  │         │ 全部专家评分提交      │
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │ summary  │ 组长汇总      │
                │  │    │设基准价/生成报告│          │
                │  │    └────┬─────┘               │
                │  │         │ 组长生成报告+签名     │
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │pre_award │ 待定标        │
                │  │    │ 候选人公示│               │
                │  │    └────┬─────┘               │
                │  │         │ 确认中标             │
                │  │         ▼                     │
                │  │    ┌──────────┐               │
                │  │    │ awarded  │ 已定标        │
                │  │    │ 中标公示  │               │
                │  │    └────┬─────┘               │
                │  │         │ 合同归档/跳过        │
                │  │         ▼                     │
                │  │    ┌──────────┐◀──────────────┘
                │  │    │ archived │ 已归档（终态）
                │  │    └──────────┘
                │  │
                │  │ 任意 bidding 之后状态
                │  ▼
                │ ┌──────────┐
                └─│ abnormal │ 异常（终止/中止/重新招标）
                  └────┬─────┘
                       │ terminate -> archived（废标）
                       │ suspend  -> 冻结，可恢复回原状态
                       │ retrial  -> 回到 draft（保留历史）
                       ▼
                  （见 §7 异常分支）
```

**状态→操作人→动作** 详见 `04-role-state-action-matrix.md`。

### 关键状态约束

| 状态 | 可见页面 | 关键约束 |
|---|---|---|
| draft | 招标文件编辑 | 可改 scoreItems；发标前 |
| bidding | 公告已发布 | 招标文件 scoreItems 冻结；可发变更公告 |
| bid_open | 投标人工作台 | 可报名/缴费/上传/报价；截止前 |
| pre_open | 项目跟踪 | 投标文件 submitStatus 锁定（截止后不可改） |
| opening | 开标大厅 | 仅此状态可解密；解密动作受 §4 矩阵约束 |
| evaluating | 评标大厅 | 专家评分（仅 owner 评自己 task）；全提交后进 summary |
| summary | 评标大厅 | 仅组长可设基准价/汇总/生成报告 |
| pre_award | 定标页 | 候选人公示；可跳过公示 |
| awarded | 中标通知书 | 发送中标通知书 |
| archived | 归档 | 终态；只读 |

## 2. Notice（公告）状态机

```
┌──────┐  发布   ┌──────────┐  变更   ┌──────────┐
│draft │────────▶│published │────────▶│ changed  │
└──┬───┘         └────┬─────┘         └──────────┘
   │                  │ 撤回
   │                  ▼
   │             ┌──────────┐
   │             │withdrawn │
   │             └──────────┘
   │                  │
   └──────────────────┴────────▶ ┌──────────┐
                                 │ archived │（标段归档时）
                                 └──────────┘
```

- 澄清公告（clarification）作为独立 Notice 记录，不改变原公告状态，但关联 packageId。
- 变更公告（change）必填 `changeReason`，发布后原公告状态变为 `changed`。

## 3. BidFile（投标文件）状态链 ★ P0

这是 P0-2/P0-3 的核心。状态链必须串行，缺环节阻断。

```
                  上传
                   │
                   ▼
              ┌─────────┐
              │uploaded │ 文件已上传，未签章
              └────┬────┘
                   │ 签章（必须）
                   ▼
              ┌─────────┐
              │ signed  │ 已签章，未加密
              └────┬────┘
                   │ 加密（密码 或 CA）
                   ▼
         ┌─────────────────┐
         │    encrypted    │
         ├─────────────────┤
         │ password=草稿   │──▶ 不可进入 submitted
         │ ca=可正式提交    │
         └────────┬────────┘
                  │ 提交（仅 ca 可提交）
                  ▼
              ┌──────────┐
              │submitted │ 正式提交（锁定，开标前不可改）
              └────┬─────┘
                   │ 开标解密
                   ▼
              ┌──────────┐
              │decrypted │ 已解密（开标大厅）
              └──────────┘
```

### 状态约束（验收点）

| 当前状态 | 允许动作 | 阻断条件 |
|---|---|---|
| uploaded | 签章 | - |
| signed | 加密 | 未签章不可加密 |
| encrypted(password) | 保存草稿、重新加密为 CA | **不可 submit**（P0-3） |
| encrypted(ca) | 正式提交 | - |
| submitted | 撤回（截止前）/ 解密（开标时） | 截止后不可撤回；非投标人不可解密（P0-1） |

**回执规则**：仅 `submitted` 且 `encryptMethod=ca` 生成正式回执号；密码加密保存草稿不生成回执。

## 4. OpeningSession（开标会）状态机

```
┌────────┐  主持人/监督确认  ┌──────────┐  开始签到  ┌─────────┐
│initial │─────────────────▶│ prepared │──────────▶│signing  │
└────────┘                  └──────────┘            └────┬────┘
                                                        │ 全员签到/缺员确认
                                                        ▼
                                                   ┌─────────┐
                                            ┌──────│decrypt  │ 解密阶段
                                            │      └────┬────┘
                                            │           │ 全部解密/废标处理
                                            │           ▼
                                            │      ┌─────────┐
                                            │      │ singing  │ 唱标
                                            │      └────┬────┘
                                            │           │ 结束开标
                                            │           ▼
                                            │      ┌─────────┐
                                            └──────│ closed  │ 已结束（生成记录）
                                                   └─────────┘
```

- 缺员签到时，主持人「进入开标」需二次确认（`04` 矩阵约束）。
- 解密阶段：投标人在此阶段且仅此阶段可点解密；非此状态解密按钮不出现。

## 5. Evaluation（评标）状态机

```
┌─────────┐  专家抽取/确认  ┌──────────┐  专家接受+签到  ┌─────────┐
│ initial │───────────────▶│extracting│───────────────▶│reviewing│
└─────────┘                 └──────────┘                └────┬────┘
                                                            │ 专家评分
                                                            ▼
                                                       ┌─────────┐
                                                       │ scoring │ 评分中
                                                       └────┬────┘
                                                            │ 全部专家提交评分
                                                            ▼
                                                       ┌─────────┐
                                                       │summary  │ 组长汇总
                                                       │ 组长设置基准价 │
                                                       └────┬────┘
                                                            │ 生成报告+签名
                                                            ▼
                                                       ┌─────────┐
                                                       │ closed  │ 评标结束
                                                       └─────────┘
```

- 组长（isLeader=true）独占 `summary` 状态的「统计结果」「生成报告」动作。
- 基准价类型由 TenderDoc.scoreMethod 决定（lowest/average），组长确认后生成价格分。

## 6. ExpertTask（专家任务）状态机 ★ P1

```
┌─────────┐  短信通知  ┌──────────┐  接受  ┌─────────┐  进入评标 ┌──────────┐
│extracted│──────────▶│ notified │───────▶│accepted │──────────▶│evaluating│
└─────────┘            └────┬─────┘        └─────────┘            └────┬─────┘
                            │ 拒绝                                │ 提交评分
                            ▼                                     ▼
                       ┌──────────┐                          ┌──────────┐
                       │declined  │                          │submitted │
                       └──────────┘                          └──────────┘
```

- 现状：专家直接进评标页。补全后：抽取 -> 通知（notifiedAt）-> 接受/拒绝 -> 评标 -> 提交。
- 拒绝需记录原因并触发补抽（原型中标记，不自动补抽）。

## 8. 状态机的实现约束

1. **状态字段统一**：所有实体状态用英文枚举，中文展示走 `permissions.js` 的 STATUS_COLORS 映射。
2. **非法迁移阻断**：状态机迁移必须在 store 层校验（如 `submitted` 不可回退到 `uploaded` 除非撤回），UI 层只触发、不判断。
3. **迁移留痕**：每次状态迁移写一条 AuditLog（who/when/from/to/action）。
4. **跨实体联动**：Package.opening 状态进入时，OpeningSession 才可签到；BidFile.submitted 是解密的前置条件。
