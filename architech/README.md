## 1. 为什么要重新设计

历史评审反复暴露同一类问题，根因不在「页面没做好」，而在「设计没做完」：

| 根因 | 表现 | 后果 |
|---|---|---|
| 数据模型晚于 UI | 招标公告关键信息堆在富文本；标段只有名称预算 | 开标/报价/公告无法引用结构化数据 |
| 只走主路径 | 公告只有新建表单，无草稿/变更/撤回 | 生命周期断裂 |
| 需求不可追溯 | 出现「复核人」「单一来源」等无依据字段 | 凭空增删 |
| 角色边界靠页面权限 | 权限只到「页面级」，不到「状态×动作级」 | P0：招标人能替投标人解密 |
| 状态散落组件内 | 投标文件状态停在 `useState`，不进 store | 开标读不到投标文件、版本链断裂 |

本设计的目标是**把以上五点在写代码之前补齐**，使后续每一行实现都能回溯到某个实体、某个状态、某条规则、某次验收。

## 2. 设计原则

1. **数据模型先行**：任何页面字段必须对应实体字段；禁止用富文本承载需被下游解析的结构化信息。
2. **生命周期显式化**：每个核心业务对象必须有状态机；状态决定可见页面、可执行动作、操作人、下一状态。
3. **权限到状态×动作**：页面级权限只解决「能不能进」，状态×动作矩阵解决「进来后能不能做」。
4. **需求可追溯**：每个字段/按钮/菜单可回溯到需求条目；需求未明确的，先进入「待澄清问题」而非直接实现。
5. **核心规则可验收**：评标办法驱动、加密链、版本链、串标预警等核心规则必须有可执行的验收用例。
6. **改代码先改规格**：发现需求矛盾时，先更新本目录文档，再改实现。

## 4. 核心结论（5 条）

1. **标段（Package）才是流程的主体，不是项目（Project）。**
   一个项目可含多个标段，投标/开标/评标/定标按标段独立推进。现有实现把状态挂在 Project 上，导致多标段无法独立流转。→ 见 `02`、`03`。

2. **权限模型必须从「页面级」升级为「状态×动作级」。**
   P0 解密越权的根因是：页面能进 ≠ 动作能做。解密动作的合法主体是「该标段的该投标人」，与角色无关、与状态强相关（仅开标解密阶段）。→ 见 `04`。

3. **投标文件必须建模为带版本与状态链的独立实体，并落入共享 store。**
   现状是组件 `useState`，开标大厅、评标大厅、版本链三处各读各的。→ 见 `02` §BidFile、`05` §加密链与版本链。

4. **评标办法是招标文件的「配置产物」，专家评分页是其「消费方」。**
   评分项/权重/基准价算法在招标文件发布时冻结，专家评分页必须读冻结配置，不能写死商务30/技术40/价格30。→ 见 `05` §评标办法。

5. **串标围标预警是规则引擎，不是按钮。**
   原始需求只写「自动预警」，未给判定规则。本设计给出可落地的判定维度与阈值，作为待确认规则。→ 见 `05` §串标预警、`01` 问题 4。
