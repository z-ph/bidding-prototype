## 1. 需求理解概述

平台是全流程电子化招投标系统，覆盖 6 角色、4 采购方式、7 大模块，业务主线为：

```
信息发布 → 准入注册 → 招标发标 → 投标缴费 → 线上开标 → 线上评标 → 定标公示 → 合同归档
                                                                          ↘ 异议处理 / 费用管理 / 数据分析
```

### 1.1 门户视角与信息发布主体

**门户视角**：招投标门户是「面向所有用户的统一入口」，视角不限于登录角色，分两层：
- 浏览层（交易信息/新闻/帮助/下载/联系）：含**匿名游客**，无需登录。
- 操作层（报名/下载收费文件/投标/异议）：需登录，按角色限定。

**信息发布主体**：门户是展示通道，不主动发布交易信息。发布按信息类型分属不同角色：

| 信息类型 | 发布主体 | 发布位置 |
|---|---|---|
| 招标/变更/中标候选人/中标/异常/澄清公告 | 招标人或招标代理 | 交易系统（发标/定标环节），门户同步展示 |
| 新闻公告/系统公告/政策法规 | 平台管理员 | 信息管理后台（门户后台） |

> 即：交易类公告由招标人/代理在交易系统发布、门户透出（同一数据源，见 Q14）；新闻/系统公告由管理员通过信息管理后台直接发布。两条发布路径不同，不可混淆。

> 五类内容（交易信息/新闻公告/帮助中心/下载中心/联系我们）的内容范围、发布主体、展示形态、用户操作、关联实体、验收点逐项拆解，见 [`portal-content.md`](./portal-content.md)。该详述引出实体决策：交易公告用 `Notice`（关联 packageId，招标方发布），新闻/系统公告/政策法规用 `PortalContent`（不关联标段，管理员发布），`02` 需补此实体。

### 1.2 角色（6）与行为矩阵

**角色定位**：

- **招标人**：采购发起方。创建项目、编制招标文件、主持开标、定标。
- **招标代理**：受招标人委托，在受委托项目内与招标人等价操作，但**不可创建项目**。
- **投标人/供应商**：投标方。报名、缴费、投标、报价、异议。
- **评标专家**：评标方。评分、汇总（组长）、签名。
- **监督人员**：全程只读监督 + 异常登记，不直接改业务状态（见 Q5）。
- **平台管理员**：平台运营方。准入审核、门户内容、组织、字典、日志、供应商管理、数据分析。

### 1.3 采购方式（4）及其流程差异

| 采购方式 | 公开性 | 是否需邀请 | 开标方式 | 报价轮次 |
|---|---|---|---|---|
| 公开招标 | 公告 | 否 | 开标大厅 | 1 轮 |
| 邀请招标 | 邀请函 | 是 | 开标大厅 | 1 轮 |
| 公开询比价 | 公告 | 否 | 比价大厅 | 可多轮 |
| 邀请询比价 | 邀请函 | 是 | 比价大厅 | 可多轮 |

**关键差异**：招标类走「开标大厅」一次性唱标；询比价类走「比价大厅」支持二次报价。现有实现未区分这两条路径。

## 2. 范围界定（原型边界）

本设计面向**可演示、可验收的原型**，明确以下边界：

| 在范围内 | 不在范围内（仅预留接口/沙箱） |
|---|---|
| 全角色、全流程页面与状态流转 | 真实 CA 证书签发与硬件 UKey 通信 |
| 结构化数据建模与 localStorage 持久化 | 真实短信网关、邮件网关 |
| 角色×状态×动作权限模型 | 真实支付通道（文件费/保证金/平台费） |
| 评标办法配置驱动评分 | 真实电子签章 SDK |
| 串标预警规则引擎（演示阈值） | 与财务/ERP 等第三方系统对接 |
| Mock 数据与种子场景 | 生产级高可用、灾备、等保合规 |

## 3. 待澄清问题清单

以下问题在原始需求中**未明确或存在歧义**，需需求方逐条确认。每个问题标注影响的设计模块与默认假设（如未确认则按假设推进，但标记为「待确认」）。

### A. 流程与状态

**Q1. 多标段是否独立流转？**
原始需求提到「标段（包）管理」但未说明多标段的开标/评标/定标是否独立。
- 影响：`02` 实体设计、`03` 状态机。
- 默认假设：**标段独立流转**，项目是标段的聚合容器，项目状态由所有标段状态聚合得出。

**Q2. 投标人能否只投部分标段？**
- 影响：BidFile 与标段的关联关系。
- 默认假设：**可只投部分标段**，BidFile 挂在标段下，非项目下。

**Q3. 询比价的二次报价，所有供应商是否都参与？报价是否互相可见？**
原始需求仅写「针对一次报价情况，根据需要可开启多次报价」「最低价标红显示」。
- 影响：比价大厅状态机、报价可见性权限。
- 默认假设：**所有参与供应商都收到二次报价邀请；报价金额在轮次结束后对所有参与方可见，过程中不可见。**

**Q4. 串标围标预警的判定规则是什么？**
原始需求只写「自动预警，对串标、围标情况的自检预警」，无判定维度。
- 影响：`05` 串标预警规则。
- 默认假设（待确认）：见 `05` §串标预警，采用 IP/MAC 去重 + 文件哈希 + 报价离散度三维度。**此为最需要确认的规则。**

### B. 权限与角色

**Q5. 监督人的权限边界：只能看，还是可以叫停？**
原始需求反复出现「全程监督」，但未说监督人能否中止开标/评标。
- 影响：`04` 角色状态动作矩阵。
- 默认假设：**监督人全程只读 + 可登记异常 + 可发起「叫停」（标记异常，但不直接改变业务状态，需招标人/管理员确认）。**

**Q6. 招标代理的权限与招标人是否完全等价（在受委托项目内）？**
- 影响：`04` 矩阵中代理列。
- 默认假设：**在受委托项目内等价，但无「创建项目」权限（项目由招标人创建后委托）。**

**Q7. 专家回避规则除「回避单位」外，是否包含同项目已报名投标单位？**
- 影响：`05` 专家抽取算法。
- 默认假设：**包含。抽取时过滤 avoidOrgs ∪ 本标段已报名投标人所属机构。**

### C. 数据与配置

**Q8. 评标办法的评分项是固定三类（商务/技术/价格）还是可自定义？**
原始需求写「综合评分法（最低价基准/平均价基准）、经评审最低价法、手工输入」，但评分项维度未明确。
- 影响：`05` 评标办法、TenderDoc 实体。
- 默认假设：**评分项可自定义（名称+权重），但价格项必须存在且由基准价算法自动计分；权重合计须为 100。**

**Q9. 中标候选人产生几名？公示几名？**
原始需求写「可根据报价产品分数量中标多个供应商」，但评标推荐候选人数量未明确。
- 影响：EvaluationResult、AwardCandidate 实体。
- 默认假设：**评标推荐 1~3 名候选人（招标文件可配），公示全部推荐候选人，定标从中选 1（或多）名。**

**Q10. 数据隔离的粒度？**
原始需求写「子账号业务数据显示进行数据隔离配置」。
- 影响：Organization、数据范围过滤。
- 默认假设：**四档：本人/本部门/本企业/全部，挂在组织节点上，作用于项目与公告列表。**

### D. 业务规则

**Q11. 保证金是线上还是线下？退保证金流程是否存在？**
原始需求写「线下缴纳，在线提交缴纳凭证待审核」，未提退还。
- 影响：Payment 实体、费用管理。
- 默认假设：**缴纳为线下凭证+线上审核；退还仅做状态记录（已退/待退），不实现真实退款通道。**

**Q12. 异常公告（终止/中止/重新招标）对已投标供应商和已缴费用如何处理？**
原始需求未说明。
- 影响：Abnormal 实体、状态机分支。
- 默认假设：**终止→已缴保证金标记「待退还」、投标文件标记「已废标」；中止→流程冻结可恢复；重新招标→标段状态回到「待发标」并保留历史记录。**

**Q13. 中标合同归档「可跳过」时，流程如何结束？**
原始需求写「该步骤可根据项目需要跳过」。
- 影响：定标状态机终止态。
- 默认假设：**跳过合同归档时，定标状态直接进入「已完成/已归档」，合同字段为空且标记「未归档」。**

**Q14. 门户公告与交易系统公告是否同一数据源？**
原始需求在门户模块写「发布交易信息」，又在交易系统写「招标公告发布」。
- 影响：Notice 实体唯一性。
- 默认假设：**同一数据源。交易系统发布 → 门户展示，不存在两份公告。**
