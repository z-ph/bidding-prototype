# 招投标原型交互范式评审与权限矩阵

> 评审日期：2026-07-17
> 评审范围：`src/components/Layout.jsx` 各角色菜单、`src/views/` 核心页面、现有评审意见（yy0、page-review checklist、0717 符合性检查）
> 核心结论：当前原型是**功能清单式导航**，不是**项目流程式导航**。所有角色都应把"项目/标段"作为核心对象，把阶段操作内聚到项目上下文，而不是在左侧菜单平铺功能项。

---

## 一、总体问题

1. **左侧菜单按功能模块平铺**，没有按业务流程聚合。
2. **相同阶段页面（开标大厅、评标大厅、招标文件等）被多个角色独立入口**，但都应从项目跟踪进入。
3. **一期已砍掉的功能（费用、报名、异议、合同归档、发票）仍保留在菜单中**，造成信息干扰。
4. **工作台与菜单重复**，但工作台没有替代菜单成为主操作台。
5. **项目跟踪页已经被设计成正确的时间线，但没有成为各角色的主工作页**。

---

## 二、分角色评审

### 2.1 招标人

**现状菜单（14 项）**：项目管理、采购需求、招标文件、发布公告、公告列表、供应商授权、费用管理、异议管理、确认中标人、中标通知书、合同归档、审批中心、审批流配置、采购数据分析。

**问题**：
- 菜单臃肿，远超 7±2 记忆容量。
- 招标文件、发布公告、确认中标人、中标通知书等本质上是"项目的阶段操作"，不应与"项目管理"并列。
- 费用管理、异议管理、合同归档已在一期口径中砍掉，仍保留菜单。
- 审批流配置属于系统管理，不应与日常"审批中心"并列。

**重构建议**：
- 保留主导航：工作台、待办中心、项目管理、采购需求库、审批中心、采购数据分析、消息中心、系统设置。
- 阶段操作（招标文件、公告、开标、评标、定标、通知书）全部下沉到**项目详情/项目跟踪页**。

## 三、统一交互范式原则

1. **项目是一切业务的上下文**：所有阶段页面（招标文件、开标大厅、评标大厅、定标、通知书）必须从项目进入，无 `projectId` 时 redirect 到项目列表。
2. **项目跟踪页 = 角色工作台**：每个角色看到该项目当前阶段以及自己能执行的下一步操作。
3. **角色 + 状态驱动操作按钮**：不同角色、不同项目状态显示不同操作，避免招标人看到"去上传投标文件"这类不属于他的动作。
4. **左侧菜单只保留聚合入口**：项目、审批、分析、消息、个人/企业档案、系统管理。
5. **已砍掉的功能不出现在主导航**：避免干扰。
