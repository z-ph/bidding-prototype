## 二、问题根因分析

### 1. 数据模型先于 UI 的意识不足

**表现**：
- 招标公告的关键信息（名称、编号、开标/截止时间、评标方法、开标一览表、报价字段）全部堆在富文本里。
- 标段只做了名称和预算，缺少采购方式、标书费、保证金、投标起止时间。
- 采购需求作为独立业务对象完全没有建模。

**本质原因**：
开发阶段直接从「页面长什么样」出发，而不是先定义「业务实体有哪些、字段是什么、关系如何」。富文本成了万能兜底，导致后续开标、报价、公告发布无法引用结构化数据。

### 2. 只覆盖「新建」主路径，忽略生命周期管理

**表现**：
- 公告发布只有新建表单，没有列表、草稿、撤回、变更管理。
- 变更公告没有变更原因字段。
- 缺少澄清公告类型。

**本质原因**：
对业务对象生命周期理解不完整。公告不是一次发布就结束，而是会经历草稿 → 发布 → 变更/澄清 → 撤回 → 归档的完整周期。

### 3. 需求可追溯性缺失

**表现**：
- 招标文件出现「复核人」字段，但需求文档未提及。
- 采购方式下拉出现「单一来源」，但应为「邀请询比价」。
- 采购需求模块被遗漏。

**本质原因**：
UI 实现与需求文档之间没有逐条映射。每个字段、每个选项、每个菜单项都应该能回溯到需求条目；没有追溯就容易出现「凭空增加」或「遗漏」。

### 4. 角色与流程边界未在设计阶段明确

**表现**：
- 组织方式（自行招标/委托代理）缺失，导致项目成员组成规则无法落地。
- 投标人缺少质疑招标文件的正式入口。

**本质原因**：
没有先画「谁在什么状态下能做什么」的矩阵。委托代理需要代理机构确认合同、投标人需要质疑渠道，这些都是跨角色的流程节点，必须在写代码前定义清楚。

### 5. 验收 checklist 执行不到位

**表现**：
上述问题在 build 通过、页面能打开的情况下被遗漏。

**本质原因**：
`npm run build` 只能验证语法和打包，不能验证业务正确性。缺少按角色、按状态、按数据闭环逐条核对的验收 checklist。

## 三、从开发阶段避免的规范设计建议

### 1. 先写数据模型，再写页面

- 每个业务对象必须输出字段清单、类型、必填规则、关联关系。
- 例如：项目（Project）→ 标段（Package）→ 公告（Notice）→ 投标文件（BidFile）。
- 禁止用富文本承载后续流程需要解析的结构化信息。

### 2. 为每个业务对象绘制生命周期状态机

- 公告：草稿 / 已发布 / 已变更 / 已澄清 / 已撤回 / 已归档。
- 项目：立项 / 待审核 / 报名中 / 待开标 / 开标中 / 评标中 / 已定标 / 已归档。
- 每个状态列出：可见页面、可执行操作、操作人、下一状态。

### 3. 建立需求可追溯矩阵

- 每新增一个字段、按钮、菜单项，必须标注对应需求编号或需求文档章节。
- 需求文档没有明确说明的字段，不允许直接出现在页面上；若要加，先更新需求/规格。

### 4. 角色-状态-动作矩阵（RACI 简化版）

| 业务对象状态 | 招标人 | 招标代理 | 投标人 | 专家 | 监督 |
|---|---|---|---|---|---|
| 项目立项 | 创建/编辑 | 被委托后编辑 | - | - | - |
| 报名中 | 查看报名 | 查看报名 | 报名/缴费 | - | - |
| 待开标 | 启动开标 | 协助 | 签到/解密 | 签到 | 监督 |

每个单元格明确「可查看 / 可操作 / 不可见」。

### 5. 强制 checklist 验收

- 任何页面完成前，必须按 `docs/page-review-20260714-checklist.md` 逐条核对：
  - 该页面哪些角色能访问？
  - 每个按钮在什么状态下出现？
  - 数据是否能从上游页面正确带入？
  - 操作后是否能正确进入下游页面？
- build 通过只是 checklist 的第一项，不是全部。

### 6.  living specification：改代码先改规格

- 当发现需求缺失或矛盾时，不要直接在代码里「补一个字段」或「删一个字段」。
- 先在 `spec/specs/` 或 `spec/changes/` 中更新规格，再按规格实现。
- 这样评审变更列表、提案、代码三者才能保持一致。

## 五、结论

本次 11 条评审意见集中暴露了「页面驱动开发」的通病：先画页面再补数据、只走主路径不管生命周期、字段与需求脱节。要避免同类问题复发，必须在开发阶段强制推行「数据模型 → 状态机 → 角色矩阵 → 需求追溯 → checklist 验收」的规范设计流程。
