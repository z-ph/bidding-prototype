# Proposal: 供应商授权与招标文件下载门控

## Change ID

`add-supplier-authorization-20260717`

## Why

2026-07-17 飞书云文档确认（整理稿 `docs/20260717-需求确认整理.md` §1.5、§1.12）：

- 清单 23：招标文件下载**不限报名/缴费**，但仅**经授权的供应商**可下载；**公开项目可自行下载**（§2 冲突 14）。
- 概要三：**授权周期每年一次，过期需重新授权**。
- 清单 24：**投标邀请书自动生成**，模板维护联系采购管理部。

当前原型 BidDownload 无授权门控，招标人/代理无供应商授权管理页，与新口径不符。

## What Changes

1. **供应商授权管理页**（招标人/代理）：按项目维护授权供应商名单——授权、撤销、查看授权状态与有效期。
2. **招标文件下载门控**：BidDownload 按「公开/授权」二态控制——公开项目供应商可自行下载；非公开项目仅授权名单内供应商可下载，未授权时阻断并提示；下载不以报名/缴费为前置。
3. **授权年度周期**：授权记录带年度有效期（每年授权一次），过期自动标记「需重新授权」，过期视同未授权。
4. **投标邀请书自动生成（mock）**：对被邀请/授权供应商按模板自动生成投标邀请书，项目信息、标段、时间自动填充，可在线查看；模板内容为 mock 占位（真实模板维护联系采购管理部）。

## Impact

- **新增文件**：`src/views/SupplierAuthorization.jsx`（授权管理页）、`src/data/authorizationStore.js`（授权记录 mock store，localStorage）。
- **修改文件**：`src/views/BidDownload.jsx`（下载门控与未授权提示）、`src/views/BidderProjects.jsx`（投标邀请书查看入口）。
- **spec 归属说明**：本提案新建独立 capability `authorization`，而非并入 `bidder-flow`。理由：授权是独立业务对象（有年度周期、授权/撤销生命周期），且同时被管理端（招标人/代理授权管理）与投标端（下载门控）消费；`bidder-flow` 现有需求多为报名/缴费/异议链路，`remove-deprecated-flows-20260717` 正对其做 REMOVED/MODIFIED，并入会造成两个提案的 delta 互相纠缠；独立 capability 便于归档合并与追溯。
- **技术约束**：React 18 + AntD 5 原型，mock 数据优先，外部接口只做预留（邀请书真实 PDF 生成与电子签章不实现）；页面组件放 `src/views/`，路由文件只放元数据；实施时共享文件（`Layout.jsx`、`permissions.js`、路由）由基础设施 agent 统一接线。

## Out of Scope

- 「授权供应商」与公开/邀请招标的关系确认（清单 23 追问，整理稿 §4）——本提案按「公开项目自行下载、非公开项目授权下载」二态先行落地。
- 投标邀请书真实模板内容、PDF 生成与电子签章。
- 与 `remove-deprecated-flows-20260717` 的衔接：报名/缴费下线后下载门控亦不依赖报名缴费状态（本提案门控本来就与报名缴费无关），冲突时以 2026-07-17 云文档口径为准。
