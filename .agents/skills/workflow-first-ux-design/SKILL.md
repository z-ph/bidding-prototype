# 工作流优先的 UX 设计原则

## Overview

`AGENTS.md`「工作流优先的交互设计原则」共 6 条，其中 5 条属于 UX/交互设计范畴，收录于本 skill 并附正反例（引用仓库真实代码与评审记录，方便复核对照）。

第 6 条「共享文件变更需整合审批」是工程协作治理规则，不是 UX 原则，**不在本 skill 范围**，由 `AGENTS.md` 直接维护。同理，版本号/changelog 维护、并行文件隔离等工程规则也不进本 skill。

## 原则与正反例

### 1. 正向推导：需求 → 工作流 → 页面/菜单

**原则**：先确认需求属于哪个角色的哪个业务阶段，再推导完成任务的典型路径，最后才决定需要哪些页面/菜单项。评审/反馈处理必须走「需求层 → 工作流层 → 页面层」三层推导，只有确认是页面层问题才直接改页面。禁止拿现有页面反推工作流来解释它。

- **正面**：0717 费用口径变更（`src/views/ReviewChangeList.jsx` 条目 `0717-rm-004`）：需求层（甲方确认不缴费、支付需求作废）→ 工作流层（投标流程去掉缴费前置）→ 页面层（删 `BidPayment`/`BidderInvoices` 页面/路由/菜单/权限，`FeeManage` 改造为台账）。先改口径，再改流程，最后动页面。
- **反面**：收到「某页面缺个按钮」就直接加，不问该需求是否已被新口径作废——对照 `cxy-013`：旧评审要求补缴费入口，新口径下只能标「无需修复」，直接改页面等于白改。或「已经有这个页面，所以菜单里得有它的入口」——用页面存在性反推菜单，菜单退化成页面清单。

### 2. 项目是一切业务的上下文

**原则**：阶段页面（招标文件、开标大厅、评标大厅、定标、通知书等）必须从项目跟踪/项目列表带着 `projectId` 进入，禁止空载进入；无 `projectId` 时重定向到对应角色的项目列表。

- **正面**：`src/views/OpeningHall.jsx:53-55` 无 `projectId` 时不渲染业务内容，返回 `<ProjectEntryGuard />` 引导回项目列表；阶段间跳转保持上下文传递，如开标完成进入评标：`navigate({ to: '/admin/evaluation-hall', search: { projectId } })`（`OpeningHall.jsx:281`）。
- **反面**：阶段页面空载渲染、写死演示项目。真实案例见 `ReviewChangeList.jsx` 条目 `1415-007`：「确认中标人/中标通知书/合同归档仍写死单一项目，未按项目维度展示」——页面能打开，但永远是同一个假项目，业务上等同于错误。

### 3. 菜单按工作流聚合，禁止平铺

**原则**：左侧菜单只保留按业务域聚合的入口；「能访问的页面」不等于「应该出现在主导航的入口」；每加一个功能不得 append 一个顶层菜单项。

- **正面**：`src/components/Layout.jsx` 按业务域分组而非按页面平铺——「项目管理」分组聚合 项目列表/创建项目/项目跟踪（`Layout.jsx:60-67`），「组织与用户」分组聚合 用户权限/组织机构/子账号管理（`Layout.jsx:132-138`）。
- **反面**：`docs/role-permission-matrix.md:23` 记录的旧现状——招标人菜单曾平铺 14 个顶层项；同一角色出现名称不同但职责重叠的两个入口（如「工作台」与「管理控制台」并列）；点进去只有几个跳转按钮的「空壳页面」。

### 4. 操作由角色 + 状态驱动

**原则**：页面上的按钮由当前角色和业务对象状态共同决定。权限表决定「能不能进这个页面」，角色 + 状态决定「进来之后能点哪个按钮」，两个条件缺一个判断都不算做对。

- **正面**：`src/views/ProjectDetail.jsx:206-208`「发标」按钮同时校验角色与状态：`role === 'tenderee' && project.status === 'draft'` 才渲染——投标人、监督员看不到（角色条件），发标后按钮消失（状态条件）。「编辑」按钮受 `beforeOpen`（`draft/tendering/registering`）约束（`ProjectDetail.jsx:123`）。
- **反面**：角色错位——招标人也看得到「上传投标文件」按钮（上传标书是投标人的动作）；状态错位——项目已开标、已定标后仍挂着「发标」「编辑招标文件」按钮，点下去要么报错要么破坏流程。

### 5. 已砍掉/未实现的功能不出现在主导航

**原则**：一期已砍掉或仅预留接口的功能（报名、异议、合同归档、发票等），不得出现在左侧主导航。

- **正面**：0717 新口径整改后（`0717-rm-004`，commit `feat(remove-deprecated)`），`BidPayment`/`BidderInvoices` 页面/路由/菜单/权限整体删除；当前 `Layout.jsx` 各角色菜单中已无报名、异议、合同归档、发票入口（可全文检索确认）。注意「费用台账」不是违规残留——它经 0717 新口径改造为「中标人投标费用登记台账」（线下收缴、无在线支付），是甲方确认后的新功能。
- **反面**：`docs/role-permission-matrix.md:13` 记录的旧问题——「一期已砍掉的功能（费用、报名、异议、合同归档、发票）仍保留在菜单中，造成信息干扰」。用户点进去发现是空壳或死功能，对系统信任度下降。

## Quick Reference

| 场景 | 检查点 |
|---|---|
| 新增页面/菜单项 | 能否正向推导到需求工作流的明确步骤？现有页面能否承载，而非新开？ |
| 评审/反馈处理 | 先判需求层 → 工作流层 → 页面层；只有页面层问题才直接改页面 |
| 阶段页面 | 必须带 `projectId` 进入；无 `projectId` 重定向到项目列表，禁止空载渲染 |
| 页面按钮 | 角色 + 业务状态双条件都判断了吗？ |
| 菜单改动 | 按业务域聚合，不 append 顶层项；砍掉的功能不进主导航 |

## 复核引用

红线审核发现违规时，可引用本 skill 对应条目作为判定依据；登记 `ReviewChangeList.jsx` 时 `fix` 字段可引用对应条目说明整改口径。新增经评审确认的真实案例（尤其反面案例）时，补充到对应原则小节，保持「原则—案例」可追溯。
